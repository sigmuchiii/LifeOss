# Открытые вопросы LifeOS v4

Создано: 2026-05-05
Обновлено: 2026-05-28
Статус: активный список нерешённых вопросов

Этот файл содержит только вопросы, на которые ещё нет ответа или есть только частичный ответ. Подтверждённые решения должны быть перенесены в [decisions-log.md](decisions-log.md), спецификации модулей в [modules/](modules/) или в [future-features.md](future-features.md), если идея явно отложена.

Ответы владельца могут быть свободными. Нужно интерпретировать весь ответ по смыслу, а не только прямое `да` или `нет`.

Статусы:

- Открыт: нужен ответ.
- Частично: часть решения уже есть, но важная неопределённость остаётся.

## Релиз

194. Partial. Владелец работает как соло-разработчик без отдельного юрлица/компании. Private dogfood-сборки могут оставаться неподписанными или использовать явно задокументированную заглушку издателя. Для публичного Windows-релиза нельзя придумывать юридическое имя: `LIFEOS_PUBLIC_PUBLISHER` должен совпадать с subject/publisher реального индивидуального code-signing certificate владельца либо будущего юрлица. Release pipeline уже умеет dry-run без сертификата и блокирует public production как `NotSignedBlocked`; открытым остаётся только точное значение subject/publisher, когда будет выбран реальный сертификат.

## Физическое Здоровье

102. Partial. Раздел пока не уточняется дальше, потому что активная работа над ним отложена. MVP-интерфейс сохраняет зоны тела, заметки, жалобы, контекст/триггеры, тип врача, дату/время приёма, информацию о визите и имена вложений в основном SQLite-хранилище `physical_zones`. Остаются открытыми будущая медицинская модель за пределами body-zone MVP, shared attachment storage, severity/history taxonomy и возможный Markdown/export формат для врачебных заметок.

## Мобильная Готовность

177. Partial. Мобильная версия отложена. Решение владельца: к началу мобильной версии все модули должны быть приведены в идеальное состояние, а первая мобильная версия должна включать все модули, которые существуют в приложении на момент старта мобильной разработки. Открытой остаётся глубина чтения/записи и мобильная форма работы для каждого модуля.

178. Partial. Мобильная запись не входит в текущий private dogfood desktop scope. Для будущей первой мобильной версии остаётся выбор: разрешать ли офлайн-запись до полноценной синхронизации, или сначала делать мобильные экраны только для чтения/ограниченной записи до появления метаданных устройств, правил конфликтов и записей об удалениях?

179. Open. Вопрос отложен до старта мобильной версии. Какие правила локального времени и часовых поясов нужны для расписаний на нескольких устройствах: текущее локальное время каждого устройства, сохранённый часовой пояс записи, часовой пояс устройства-создателя или настройка на уровне календаря?

180. Partial. Текущий desktop `change_log` фиксирует локальные изменения до появления cloud providers, но ещё не является multi-device sync metadata contract. До мобильной записи остаётся выбрать обязательный минимум: id устройства/клиента, глобально стабильный id операции, ревизия записи, база конфликта, срок хранения записей об удалениях и обработка расхождения часов.

183. Open. Вопрос отложен до старта мобильной версии. Как должен развиваться контракт каналов уведомлений от текущего `windows_system` к нейтральной модели для Windows-десктопа, мобильных локальных уведомлений, мобильных push-уведомлений и будущего Telegram?

184. Open. Вопрос отложен до старта мобильной версии. Каждое устройство должно запускать свой локальный планировщик напоминаний, или синхронизация/привязка устройств должна выбирать один главный источник доставки, чтобы избежать дублей?

## Система, Данные И Реализация

129. Partial. Основное хранилище решено для Задач, событий Календаря, напоминаний, привычек, сессий Фокуса, настроек, Заметок, Быстрых заметок, Финансов, Проектов, Контактов, Дневника, Физического здоровья и Трейдинга. Для Дневника текущий MVP-контракт медиа закрыт как локальное attach/preview/rename/delete вместе с записью дня. Открытой остаётся общая durable-модель вложений/медиа для модулей за пределами Notes и текущего Diary MVP, а также будущие правила синхронизации этих файлов.

131. Partial. Текущие Настройки покрывают тему, язык, время подъёма/сна, видимость модулей, переключатели уведомлений, статус синхронизации, экспорт данных, отдельные ручные варианты `Резервная копия LifeOS` и `Резервная копия Obsidian`, поиск, проверку целостности, dry-run и apply восстановления/импорта, pre-update rollback entry point и импорт Obsidian. Остаются открытыми: долговременный порядок модулей, настройка каналов уведомлений и будущие настройки автоматического применения восстановления/импорта.

132. Partial. Первая десктоп-структура папок в основном решена: собранная Windows-версия использует `<install directory>/data` как корневую папку данных, внутри лежат `lifeos.sqlite3`, `notes/`, `notes/_attachments/`, `diary/`, `exports/lifeos-export-*`, `backups/lifeos-backup-*`, `backups/obsidian-backup-*`, `backups/pre-update/lifeos-pre-update-v*`, `search/index.sqlite3`, legacy/frontend fallback-записи в `frontend-records.json`, фронтенд-настройки в `frontend-preferences.json`, `startup-failure.json` при failed upgrade и маркер runtime-версии. Новые версии, установленные в ту же папку, используют те же `data`, а запуск релизной сборки создаёт резервную копию перед обновлением до открытия SQLite/migrations, если данные пришли из более старой версии. Complete pre-update artifact можно применить через безопасный restore/import flow, а после отката безопасна версия из `source_app_version` или более поздний исправленный build. Dev/debug может продолжать использовать стандартную папку данных Tauri. Открытыми остаются папки вложений вне Notes.

133. Open. Какие правила конфликтов нужны, если Markdown-файлы или файлы вложений редактируются вне LifeOS, пока приложение запущено?

134. Partial. Структурированные записи используют стабильные строковые ID, а читаемый экспорт сохраняет текущие ID/Markdown-пути. Артефакт восстановления/импорта версии 1 проходит dry-run перед применением: сохраняет ID, блокирует опасные/невалидные артефакты, проверяет связи внутри артефакта и не переназначает ID/ссылки. Apply для v1 поддержан через backup-before-restore, staging, замену текущих данных и финальную проверку целостности; complete pre-update artifacts поддержаны как rollback-only источник. Остаются открытыми явный дизайн таблицы переназначений для более поздних версий артефактов и пользовательский UI конфликтов за пределами текущего отчёта о блокировке.

135. Open. Какие тестовые/демо-данные должны существовать для дизайна, скриншотов, ручного QA и разработки?

137. Partial. Текущая гипотеза: все MVP-модули должны работать офлайн без внешних сервисов. Остаётся вопрос: может ли любая будущая функция синхронизации или мобильная функция требовать режим только онлайн, или офлайн-первый подход должен оставаться обязательным везде?

167. Partial. Заметки используют `notes/_attachments/` и сохраняют импортированные относительные пути вложений. Дневник в текущем MVP использует локальный attach/preview/rename/delete контракт внутри записи дня, но это не является общей file/media-моделью. Общая стратегия медиа для Задач, Физического здоровья, Трейдинга, будущих общих файлов и будущих формальных Diary media rules остаётся открытой: нужен app-wide `attachments/{module}/{record}/` store или отдельные папки/правила по модулям?

196. Partial. Финансы имеют основное хранилище для счетов, транзакций и бюджетов, но интерфейс бюджетов, категории, повторяющиеся платежи, типы наличных/карточных счетов, дополнительные валюты и связи Финансов с Трейдингом или Проектами всё ещё отложены или недоопределены.

202. Partial. Reminder backend уже поддерживает самостоятельные напоминания: create/list/due paths, recurrence-aware due occurrences, delivery attempts/statuses, Home cards и browser fallback. При этом видимый create/edit UI для самостоятельных напоминаний ещё не закреплён: оставить их только как backend/Home surface до будущего модуля `Напоминания`, добавить минимальный create/edit flow в Home/Settings, или продвинуть отдельный модуль `Напоминания` раньше?

203. Partial. Для текущего сценария восстановления после переустановки Windows Google Drive flow решён: пользователь подключает Google account, задаёт название папки, LifeOS синхронизирует encrypted backup каждые 5 минут, при выходе ждёт `synced`, использует managed Google Drive key без ручного recovery key, не показывает ссылку на папку, `Хранить последних копий` и close-time toggle, а Obsidian folders входят в encrypted backup artifact. Открытым остаётся только будущий вариант, если LifeOS позже захочет выбирать существующие внешние Drive folders: нужен Google Picker с `drive.file`, широкий scope, или отдельный отказ от такого сценария?
