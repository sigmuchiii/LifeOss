## Что это за проект

LifeOss — персональная, локальная (local-first) Windows-десктоп «операционная
система жизни» для одного владельца (werty4870 / sigmuchiii). Не публичный
продукт. Русский язык интерфейса — основной.

Проект строится **с нуля** по документации: старые доки в `docs/` описывают
зрелое предыдущее приложение (решения до 2026-06-29) — они источник продуктовых
требований, но НЕ описывают текущий код. Текущий код описан ниже и в свежих
записях `decisions-log.md` (от 2026-07-04 и позже).

## Правила работы с владельцем (обязательны)

1. Всё, что владелец пишет в чате — истина. Любое его решение фиксируется в
   `docs/decisions-log.md` в той же рабочей сессии.
2. Владелец предпочитает **целые модули за один проход** по спеке, затем
   присылает пачку правок (скриншоты/текст). Не чинить по одной мелочи.
3. Конфликты документации решаются так: свежие записи `decisions-log.md` >
   `docs/modules/*` > `docs/spec-draft.md` > `docs/archive/*`.
4. `docs/open-questions.md` — вопросы сознательно отложены, не решать их
   самовольно.
5. Отвечать владельцу по-русски.

## Рабочий процесс (workflow)

- Агент пишет код и коммитит в `main` через GitHub API (репозиторий
  `sigmuchiii/LifeOss`, публичный). Прямого доступа к машине владельца нет.
- Владелец собирает локально: `git pull` → `npm run tauri dev`
  (путь у него: `C:\Projects\LifeOss\LifeOss`), присылает ошибки и скриншоты.
- Агент не может сам собрать/запустить Tauri — проверяй код внимательно до
  коммита (TypeScript строгий, Rust без warnings по возможности).

## Backend: команды Tauri

`app_status`, `settings_get/set`,
`tasks_snapshot/add/toggle/update/delete/restore`, `trash_clear`,
`lists_add/delete`, `subtasks_add/toggle/delete`,
`habits_list/add/delete`, `habit_marks_range`, `habit_mark_set`,
`qnotes_list/add/update/delete`, `focus_add`, `focus_today`.

Все структуры сериализуются в camelCase (`#[serde(rename_all = "camelCase")]`).
Фронт вызывает через `invoke()` c camelCase-аргументами (Tauri сам мапит в
snake_case параметры команд).

## Миграции

`lifeos-storage/src/lib.rs`, массив `MIGRATIONS: &[(&str, &str)]` — id + SQL,
применяются по порядку один раз (таблица `schema_migrations`). Применены:
0001 settings, 0002 tasks, 0003 tasks_full (списки/подзадачи/статусы),
0004 habits + habit_marks + quick_notes + focus_sessions.
Новая миграция = новый элемент в конец массива. НИКОГДА не менять старые.

## Дизайн-система

- Токены в `tokens.css`; темы через `html[data-theme]` (light/dark/midnight/
  forest/sunset). Тема выбирается ТОЛЬКО в Настройках (не в топбаре — решение
  владельца).
- Карточки `.card`, кнопки `.btn-acc`/`.btn-quiet`/`.linkish`.
- **Все поля ввода** — класс `.field` (контур, мягкий фон, фокус-кольцо).
  Новые модули обязаны использовать `.field` для input/textarea/select.
- Выбор даты — общий `components/DatePicker.tsx` (не нативный input[type=date]).
- Анимации: fadeUp контента, popIn поповеров, checkPop галочек; уважать
  `prefers-reduced-motion` (уже глобально).
- Эмодзи в приоритетах/статусах задач — по вкусу владельца, сохранять стиль.
- Сайдбар: тултипы через `data-tip`, НЕ через `title`.

## Windows-грабли (уже исправлено, не ломать)

- Vite слушает `127.0.0.1` (IPv6-only binding вешает Tauri).
- Vite watcher игнорирует `src-tauri` (иначе EBUSY-краши).
- `src-tauri/icons/icon.ico` обязателен для tauri-build.
- Прерванная сборка может испортить `src-tauri\target` — лечится удалением
  папки и пересборкой (сказать владельцу).
